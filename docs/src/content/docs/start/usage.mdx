## Run it

```bash
# TUI mode (auto-detected when stderr is a TTY)
polygit [DIR]

# Default DIR is the current directory
polygit
```

`polygit` scans `DIR` **recursively** for git repositories (pruning hidden /
`node_modules` / `vendor` / `target` / `*.worktrees` dirs, and never descending into a found
repo), then fast-forwards each in parallel as it's discovered. Point it at a whole tree:

```bash
polygit ~/projects     # crawl every project
polygit ~ --depth 4    # crawl home, capped 4 levels deep
polygit --no-recursive # legacy single-level scan (immediate children only)
```

Render the result as a collapsible [directory tree](../../guides/tree-view/) with `v t`, or as
named [groups](../../guides/groups/) with `v g`. The list title shows live concurrency
(`⇄ active/cap`); if a remote rate-limits the run, `polygit` backs off automatically (see
the throttle banner) and retries with exponential backoff.

## Plain / headless mode

When stdout isn't a TTY — or with `--no-tui` — `polygit` streams plain output that's
byte-compatible with the original bash reference, so it drops cleanly into scripts and CI.

```bash
polygit --no-tui ~/projects
```

## Concurrency

Pulls run in parallel up to a limit (default: number of CPUs).

```bash
polygit -j 8 ~/projects        # flag
PULL_JOBS=8 polygit ~/projects # environment variable
```

## Timeout

Each individual pull is capped (default: 30 seconds) so one hung remote can't stall the run.

```bash
polygit --timeout 60 ~/projects
PULL_TIMEOUT=60 polygit ~/projects
```

## The two panes

- **Left `[1]`** — the repo list with a status glyph, name, and current branch. Optional
  [columns](../../guides/columns-and-glyphs/) add ahead/behind, dirty, last-commit,
  worktree, branch, and stash counts, and a clickable column header lets you sort. With
  `z`, the list renders as named [group sections](../../guides/groups/).
- **Right `[2]`** — a live log of the selected repo's pull output, or (with `i`) an info
  panel, or (with `d`) the working-tree / last-pull diff.

`Tab` (or `1`/`2`) switches focus; the active pane shows a bright rounded border. Drag the
divider between them (it shows a grip and highlights while dragging) or use `[` / `]`.

<Aside type="tip">
Press `?` at any time for the in-app help modal — five tabs (**Hotkeys**, **CLI**,
**Legend** with every glyph in both icon sets, **About**, and **Design**) switched with
`Tab`. The About tab groups its clickable links (polygit, lazygit, and a collapsible Notes group)
and shows titles only — hover a link to preview its URL in the bottom-left corner. The **Design
System** tab carries the Theme/Background/Contrast/Selection radios (click to set, click the active
value to cycle) above a live swatch showcase of the palette's semantic colors.

## Settings

`,` opens a small settings modal (persisted in `~/.config/polygit/state.json`), organized into
a **General** section (panel padding, grouping, tree view), a **Theming** section (icons, theme,
background, contrast, selection), a **Sync** section (auto-pull policy — see below), and an
**Interaction** section (hover effects), and a **Layout** section (borders). By default it uses a **tabbed** layout (sections as
vertical tabs — `←`/`→` or `Tab` switch, `↑`/`↓` move within a tab); press `v` to cycle the layout
**tabbed → accordion → flat**. In the **accordion** layout every section stacks under a collapsible
header (click a header — or `←`/`→` on the selected row's section — to fold/unfold it; a top
`[- collapse all]` / `[+ expand all]` button folds them all at once; collapsed sections persist).
The **flat** layout stacks every section expanded:

- **Panel padding** — a 1-cell inner padding inside every bordered panel and modal.
- **Icons** — switch the status / column / marker glyphs between the default Unicode set and
  an emoji set.
- **Hide zeros** — hide zero-count column cells (a dim `0` becomes blank), so only repos that
  actually have ahead/behind/dirty/worktree/branch/stash counts show a number. Emoji mode always
  hides zeros, so this row is force-selected and inert while emoji icons are active.
- **Theme** — `dark` / `light` paint a full explicit palette (background, text, and every
  accent color) so the app looks identical in any terminal. `auto` detects whether the
  terminal background is dark or light at startup (OSC 11 query, then `COLORFGBG`, then the
  Windows light/dark setting under WSL, then the macOS appearance setting) and applies the
  matching palette. Under `auto`, polygit also **re-detects at runtime** (via the tty-safe
  sources — `COLORFGBG` / the WSL registry / macOS appearance), so toggling your OS between
  light and dark re-themes the running app within a couple of seconds, no restart needed.
- **Background** — `normal` / `soft` / `terminal`. `normal` and `soft` paint a surface tone;
  `terminal` paints **no** base background, letting your terminal's own background show through
  (selected rows and modal shadows still get a tone so they stay visible).
- **Contrast** — `normal` / `soft`. Independent of Background: it narrows the text/background
  distance and desaturates the accent + semantic colors. The two axes compose freely.
- **List selection** — how the selected row is highlighted everywhere a row is selectable (repo
  list, repo page, diff file list). `blue` is a solid blue bar with white text; `subtle` is a soft
  blue tint that keeps each column's own color (status hue, branch accent) readable. Hovering the
  selected row deepens the tint in either mode so it stays distinct from a plain hover.
- **Button hover** (Theming, right below List selection) — how *buttons* (footer/modal hint chips,
  tabs, radio chips, close buttons, keyboard keys) highlight under the cursor: `subtle` (default) is
  a soft tint, `inverted` is reverse-video so the hovered button reads as a solid chip. Independent
  of List selection.
- **Grouping** — render the list as named [group sections](../../guides/groups/) (same as `z`).
- **Auto-pull on launch** — `on` / `off`. When off, launches never pull; the list shows cached
  status and you pull on demand (`e` / `E`).
- **Auto-pull limit** — `50` / `100` / `250` / `∞`. Skip the launch auto-pull when more repos
  than this are discovered (so pointing polygit at a huge tree doesn't kick off hundreds of pulls).
- **Auto-pull in tree view** — `on` / `off` (default off). The directory-tree view suppresses
  the launch auto-pull unless you turn this on.

The modal is fully mouse-driven too: click a radio chip to set that value; click the
already-active chip (or the row label) to cycle to the next value (3-radio settings like Theme
cycle left→right and wrap); and close with the `[x]` button or a click anywhere outside. The same
applies to every modal (help, confirm, copy menu, diff).

## Auto-pull on launch & the status cache

polygit is a polyrepo **status** dashboard first and a pull tool second. Every launch is useful
immediately: the list is seeded from a persisted per-repo **status cache**
(`~/.config/polygit/status-cache.json`), so each repo shows its last-known status the instant it's
discovered — no git work required. Cached-but-not-pulled-this-session repos render **dim with a
relative age** (e.g. `up-to-date 2d`, shown in the optional `t u` status column and the `i` info
panel); pulling brightens them to the live result.

Whether a launch **auto-pulls** is governed by the Settings → **Sync** section above. By default
polygit auto-pulls small flat sets (the familiar behavior) but **skips** the auto-pull when there
are more than 100 repos, or when the [directory-tree view](../../guides/tree-view/) is active —
exactly the cases where pulling everything on startup is expensive or unwanted. When a launch
skips auto-pull, the Result row reads "Ready — auto-pull off"; press **`E`** to pull every repo or
**`e`** to pull just the selected one. Either way the cache is refreshed as repos are pulled.

## New-build reload

polygit watches its own binary while running. When a newer build lands at the same path
(e.g. `make install`'s atomic rename), a notice appears in the top-right corner with a
`[reload]` button — clicking it restores the terminal and `exec`s the new binary with the same
arguments, so the fresh build takes over in the same terminal and re-runs the pulls. `[x]`
dismisses the notice until the binary changes again. The notice rides on top of every screen
(the repo list, the repo page, and any open modal), so a freshly-installed build is always one
click away.

Clicking the `built … ago` tag in the status bar opens a **Build info** modal showing the
running version, the watched executable path, when it was built, how the watch works, and
whether a newer build is waiting. `r` (a clickable hint in the bottom border, next to `esc
close`) exec-restarts into the latest build; any other key or click closes it.

## Reading the result

When the run finishes, a **Result** summary row reports how many repos were updated, were
already up to date, were skipped, had no upstream, or failed. The process
[exit code](../../reference/exit-codes/) reflects the outcome so scripts can branch on it.

A branch with no upstream configured is a distinct **no-upstream** state (glyph `▽`), not a
failure — it's kept off the Errors page and counts as done. A branch that tracks a remote ref
which no longer exists (its PR was merged and the branch deleted, so git pull reports "no such
ref was fetched") is treated the same way — nothing to pull, not a red failure.

If any repo fails, a dynamic **Errors (N)** row appears right after Result; select it to see
each failed repo with the tail of its error output. A failed pull is retried once automatically
before it's marked failed, so transient network or lock hiccups usually clear themselves.
Permanent errors — repository not found, authentication/access failures, a diverged branch
that can't fast-forward — skip the retry and fail immediately, since retrying can't change
the result. The optional [status column](../../guides/columns-and-glyphs/) (`t` `u`) names
the failure kind right in the list.
