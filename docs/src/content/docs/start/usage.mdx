## Run it

```bash
# TUI mode (auto-detected when stderr is a TTY)
polygit [DIR...]

# No args → scans the current directory
polygit
```

`polygit` scans each `DIR` **recursively** for git repositories (pruning hidden /
`node_modules` / `vendor` / `target` / `*.worktrees` dirs, and never descending into a found
repo), then fast-forwards each in parallel as it's discovered. A `DIR` that is **itself a git
repo** is shown as that single repo. Point it at a whole tree, several folders, or one repo:

```bash
polygit ~/projects                 # crawl every project
polygit ~/work ~/oss ~/some-repo   # multiple folders (the last is a single repo)
polygit ~ --depth 4                # crawl home, capped 4 levels deep
polygit --no-recursive             # legacy single-level scan (immediate children only)
```

### Named workspaces

A launch is **ad-hoc by default** — `polygit` (or `polygit DIR...`) scans the cwd / the given
folders and saves nothing. To keep a curated multi-folder set, use a **named workspace**:

```bash
polygit -w work ~/projects/a ~/projects/b   # define (or redefine) workspace "work" and open it
polygit -w work                             # reopen workspace "work"
polygit ws                                  # interactive picker of saved workspaces
polygit ws ls                               # (alias: polygit workspaces list) print them
```

Workspaces are stored in `~/.config/polygit/state.json` under arbitrary names, and the folder
picker (`A`) / remove-root (`X`) edit the **active** workspace (the one you opened with `-w` or
the picker) — ad-hoc sessions don't persist folder changes. A new name with no folders starts
from the cwd, ready for the picker. With multiple folders, the [tree view](../../guides/tree-view/)
shows each as its own top-level node (a forest).

Render the result as a collapsible [directory tree](../../guides/tree-view/) with `v t`, or as
named [groups](../../guides/groups/) with `v g`. The list title shows live concurrency
(`⇄ active/cap`); if a remote rate-limits the run, `polygit` backs off automatically (see
the throttle banner) and retries with exponential backoff.

## Plain / headless mode

When stdout isn't a TTY — or with `--no-tui` — `polygit` streams plain output that's
byte-compatible with the original bash reference, so it drops cleanly into scripts and CI.

```bash
polygit --no-tui ~/projects
```

## Concurrency

Pulls run in parallel up to a limit (default: number of CPUs).

```bash
polygit -j 8 ~/projects        # flag
PULL_JOBS=8 polygit ~/projects # environment variable
```

## Timeout

Each individual pull is capped (default: 30 seconds) so one hung remote can't stall the run.

```bash
polygit --timeout 60 ~/projects
PULL_TIMEOUT=60 polygit ~/projects
```

## The panels

polygit has up to four focusable panels, each with a stable number in its title:

- **`[1]` list** — the repo list with a status glyph, name, and current branch. Optional
  [columns](../../guides/columns-and-glyphs/) add ahead/behind, dirty, last-commit,
  worktree, branch, and stash counts, and a clickable column header lets you sort. With
  `z`, the list renders as named [group sections](../../guides/groups/).
- **`[2]` info** — the **info panel** (`i`, top of the right pane): repo facts, clickable
  branch/commit/remote/PR links, and whole-line copy on the Path / Worktrees / non-link Branch
  rows (each marked by a `⧉`).
- **`[3]` result/log** — the **result/log panel** (`I`, bottom of the right pane): a live log
  of the pull output, or with `d` the working-tree / last-pull diff. When both `[2]` and `[3]`
  show, a **draggable boundary** splits them (set by `preview_split_ratio`); hide one and the
  other fills the right pane. Both toggles are persisted, with clickable `i info` / `I log`
  footer hints.
- **`[4]` repo page** — the [repo page](#) for the selected repo (opened with `Enter`). By
  default it's **restored**: a docked panel across the bottom, so the list above stays live
  (selecting a repo retargets the panel — master-detail). Press `m` (or the `▢` title-bar
  button, left of `[esc back]`) to **maximize** it full-screen; `▣`/`m` restores it. The
  state is sticky (persisted) and also settable in Settings → Layout → **Repo page**.

`Tab` / `Shift-Tab` cycle focus across the **visible** panels (hidden ones are skipped, but
numbers stay stable); `1`/`2`/`3`/`4` jump directly. The focused panel shows a bright rounded
border. Drag the divider between the list and the right pane (it shows a grip and highlights
while dragging) or use `[` / `]`; drag the info/result boundary to resize `[2]` and `[3]`, and
the restored repo page's top border to resize `[4]`.

## Settings

`,` opens a small settings modal (persisted in `~/.config/polygit/state.json`), organized into
a **Lists** section (grouping, tree view, hide folder-header lines), a **Theming** section (icons,
theme, background, contrast, selection), a **Sync** section (auto-pull policy — see below), an
**Interaction** section (hover effects), a **Layout** section (panel padding, borders, splitter,
repo-page tabs, repo page restored/maximized, branch-check), a **Tooltips** section (a master
"All tooltips" toggle plus per-area switches — footer commands, column headers, group counts,
settings rows, help links; tooltips also require Hover effects on), an **Agent** section
(which AI coding-agent the `c` key launches — claude / codex / gemini — and a "Skip permissions"
toggle that appends the agent's bypass-all-prompts flag), and a **Pull requests** section (a
**Merged PRs** toggle — off by default; when on, merged & closed PRs also show in the PR column +
info panel, not just open ones). By default it uses a **tabbed** layout (sections as
vertical tabs — `←`/`→` or `Tab` switch, `↑`/`↓` move within a tab); press `v` to cycle the layout
**tabbed → accordion → flat**. A **search box** sits at the top of the modal: press `/` to focus
it and type to filter the rows across every tab into a flat list with the matched characters
highlighted (`↑`/`↓` move through the matches, `space`/`enter` toggles, `Esc` clears the search). In the **accordion** layout every section stacks under a collapsible
header. `↑`/`↓` (or `j`/`k`) step through both the headers and the rows of expanded sections;
`space`/`enter` (or `←`/`→`) on a focused header folds/unfolds it (when a header is focused no child
row is active — the header itself is selected); clicking a header focuses **and** toggles it; a top
`[- collapse all]` / `[+ expand all]` button folds them all at once; the content scrolls when it
overflows; collapsed sections persist.
The **flat** layout stacks every section expanded. Press **`R`** (or the footer `R reset` chip) to
**reset every setting to its default** — a confirmation lists each setting that will change as an
aligned `current → default` column (new value highlighted; favorites and workspaces are untouched):

- **Panel padding** — a 1-cell inner padding inside every bordered panel and modal.
- **Icons** — switch the status / column / marker glyphs between the default Unicode set and
  an emoji set.
- **Hide zeros** — hide zero-count column cells (a dim `0` becomes blank), so only repos that
  actually have ahead/behind/dirty/worktree/branch/stash counts show a number. Emoji mode always
  hides zeros, so this row is force-selected and inert while emoji icons are active.
- **Theme** — `dark` / `light` paint a full explicit palette (background, text, and every
  accent color) so the app looks identical in any terminal. `auto` detects whether the
  terminal background is dark or light at startup (OSC 11 query, then `COLORFGBG`, then the
  Windows light/dark setting under WSL, then the macOS appearance setting) and applies the
  matching palette. Under `auto`, polygit also **re-detects at runtime** (via the tty-safe
  sources — `COLORFGBG` / the WSL registry / macOS appearance), so toggling your OS between
  light and dark re-themes the running app within a couple of seconds, no restart needed.
- **Background** — `normal` / `soft` / `terminal`. `normal` and `soft` paint a surface tone;
  `terminal` paints **no** base background, letting your terminal's own background show through
  (selected rows and modal shadows still get a tone so they stay visible).
- **Contrast** — `normal` / `soft`. Independent of Background: it narrows the text/background
  distance and desaturates the accent + semantic colors. The two axes compose freely.
- **List selection** — how the selected row is highlighted everywhere a row is selectable (repo
  list, repo page, diff file list). `blue` is a solid blue bar with white text; `subtle` is a soft
  blue tint that keeps each column's own color (status hue, branch accent) readable. Hovering the
  selected row deepens the tint in either mode so it stays distinct from a plain hover.
- **Button hover** (Theming, right below List selection) — how *buttons* (footer/modal hint chips,
  tabs, radio chips, close buttons, keyboard keys) highlight under the cursor: `subtle` (default) is
  a soft tint, `inverted` is reverse-video so the hovered button reads as a solid chip. Independent
  of List selection.
- **Grouping** — render the list as named [group sections](../../guides/groups/) (same as `z`).
- **Hide folder lines** — drop the dim dash-fill leader lines that connect a group / folder header
  name to its right-corner count, for a cleaner header row.
- **Auto-pull on launch** — `on` / `off`. When off, launches never pull; the list shows cached
  status and you pull on demand (`e` / `E`).
- **Auto-pull limit** — `50` / `100` / `250` / `∞`. Skip the launch auto-pull when more repos
  than this are discovered (so pointing polygit at a huge tree doesn't kick off hundreds of pulls).
- **Auto-pull in tree view** — `on` / `off` (default off). The directory-tree view suppresses
  the launch auto-pull unless you turn this on.

The modal is fully mouse-driven too: click a radio chip to set **that** value (clicking the
already-active chip does nothing — it never cycles); click the row label to cycle to the next value
(3-radio settings like Theme cycle left→right and wrap); and close with the `[x]` button or a click
anywhere outside. The same applies to every modal (help, confirm, copy menu, diff).

## Auto-pull on launch & the status cache

polygit is a polyrepo **status** dashboard first and a pull tool second. Every launch is useful
immediately: the list is seeded from a persisted per-repo **status cache**
(`~/.config/polygit/status-cache.json`), so each repo shows its last-known status the instant it's
discovered — no git work required. Cached-but-not-pulled-this-session repos render **dim with a
relative age** (e.g. `up-to-date 2d`, shown in the optional `t u` status column and the `i` info
panel); pulling brightens them to the live result.

Whether a launch **auto-pulls** is governed by the Settings → **Sync** section above. By default
polygit auto-pulls small flat sets (the familiar behavior) but **skips** the auto-pull when there
are more than 100 repos, or when the [directory-tree view](../../guides/tree-view/) is active —
exactly the cases where pulling everything on startup is expensive or unwanted. When a launch
skips auto-pull, the Result row reads "Ready — auto-pull off"; press **`E`** to pull every repo or
**`e`** to pull just the selected one. Either way the cache is refreshed as repos are pulled.

## New-build reload

polygit watches its own binary while running. When a newer build lands at the same path
(e.g. `make install`'s atomic rename), a notice appears in the top-right corner reading
`↺ new build installed · [^R reload] [^X]`. Press **`Ctrl-R`** (or click `[^R reload]`) to
restore the terminal and `exec` the new binary with the same arguments, so the fresh build takes
over in the same terminal and re-runs the pulls. Press **`Ctrl-X`** (or click `[^X]`) to dismiss
the notice until the binary changes again. The notice rides on top of every screen (the repo
list, the repo page, and any open modal), so a freshly-installed build is always one key away —
and both keys work from anywhere, including inside a modal.

Clicking the `built … ago` tag in the status bar opens a **Build info** modal showing the
running version, when it was built **and how long the build took**, the **binary size** + watched
path, the **settings file location** + config-dir file count, whether a newer build is waiting,
and a **collapsible tree view of `state.json`** — a format-agnostic structural-data viewer (not
JSON-specific). Objects/arrays start collapsed, each showing its child count in a faint `{N}` /
`[N]`; `j`/`k` move, `←`/`→` collapse/expand, `space`/`enter` fold the selected node, `-`/`+`
fold/unfold all (or the `[- fold all]` / `[+ unfold all]` buttons on the card header), and clicking
a node folds it. The plain mouse wheel scrolls the tree (selection untouched, web-app style) and
`Alt`+wheel moves the selection — the same model as the main list. `r` (a clickable hint in the bottom border, next to `esc close`) exec-restarts into
the latest build; `esc`, the `[x]`, or a click **outside** closes it (clicking inside is inert).

Clicking the `vX.Y.Z` tag opens the **Changelog**: every release as a collapsible accordion whose
header is the version + a time-ago (`▸ v2.60.0 · 2d ago`), with the latest two expanded. `j`/`k`
select a release, `space`/`enter` (or a header click) fold it, `g`/`G` jump. After you reload into a
newer build, a **What's New** modal pops automatically with every release since the version you last
ran. The notes are embedded from `CHANGELOG.md`.

## Reading the result

When the run finishes, a **Result** summary row reports how many repos were updated, were
already up to date, were skipped, had no upstream, or failed. The process
[exit code](../../reference/exit-codes/) reflects the outcome so scripts can branch on it.

A branch with no upstream configured is a distinct **no-upstream** state (glyph `▽`), not a
failure — it's kept off the Errors page and counts as done. A branch that tracks a remote ref
which no longer exists (its PR was merged and the branch deleted, so git pull reports "no such
ref was fetched") is treated the same way — nothing to pull, not a red failure.

If any repo fails, a dynamic **Errors (N)** row appears right after Result; select it to see
each failed repo with the tail of its error output. A failed pull is retried once automatically
before it's marked failed, so transient network or lock hiccups usually clear themselves.
Permanent errors — repository not found, authentication/access failures, a diverged branch
that can't fast-forward — skip the retry and fail immediately, since retrying can't change
the result. The optional [status column](../../guides/columns-and-glyphs/) (`t` `u`) names
the failure kind right in the list.
