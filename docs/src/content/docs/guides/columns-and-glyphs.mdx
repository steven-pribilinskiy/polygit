The list always shows three things: a **status glyph**, the repo **name**, and its current
**branch**. Everything else is an optional column you toggle on.

## Toggling columns

Press `t` to enter the column-toggle leader, then a column key. The mode stays active
(so you can flip several) until you press `Esc`.

| Key | Column | Glyph |
|-----|--------|-------|
| `u` | Status text | short label, see below |
| `a` | Ahead / behind vs upstream | `↑N ↓N` |
| `d` | Dirty (uncommitted file count) | `•N` (amber) |
| `l` | Last-commit age | relative date |
| `w` | Worktree count | `⑃N` (cyan) |
| `b` | Feature-branch count (local branches excluding `main`/`dev`) | `⑂N` (green) |
| `s` | Stash count | `≡N` |
| `p` | Commits the last pull landed | `⇣N` (green) |
| `c` | Files the last pull changed | `±N` (cyan) |

The **pulled** (`p`) and **changed** (`c`) columns report what the *most recent pull this run*
delivered — commits landed on the current branch and files touched. Repos that were already
up to date show a dim `0`; both columns stay hidden until a pull lands a delta, then remain for
the rest of the session — so a retry / refetch (which briefly clears every result) doesn't
flicker them in and out.
The full breakdown (old→new sha, line counts, new tags/branches) lives in the info panel's
**Pulled** row (`i`).

The **status** column spells the glyph out as a short label (`queued`, `running`,
`up-to-date`, `updated`, `no upstream`, `dirty`, `throttled`), and gets more specific where
the state has a known cause: a failed repo shows its failure kind — `not found` (repo deleted
or no access), `auth`, `diverged` (can't fast-forward), `not a repo`, `timeout`, `network`,
or `lock` — and a branch whose tracked remote ref was deleted shows `ref gone` instead of a
generic `no upstream`. Same colors as the glyphs.

Enabled columns persist across runs. Count columns render a **dim zero** (`⑃0`, `≡0`) rather
than a blank in the Unicode set, so the column stays recognizable; in **emoji** mode a zero cell
is hidden instead (a colorful emoji beside `0` is clutter). The dirty marker is **amber**, not red —
uncommitted changes are a "modified" state, not an error (red stays for genuine failures). A column every repo leaves empty (no
worktrees, no stashes, or `≤1` branch everywhere) auto-hides once its data has loaded, and its
`t`-menu chip goes dim and inert until some repo has a value.

## Sorting

Press `s` to enter the sort leader, then a column key — or click a column header (including
the **branch** header). The 2-row header shows a `▲`/`▼` indicator on the active sort column.
With hover effects on, dwelling ~1s over a column title (`wt`, `st`, `pr`, …) shows a tooltip
explaining it, and dwelling over a group/folder header's right-corner count (e.g. `(27)`) shows
its breakdown ("27 repos in group · 3 running · 1 failed").
The list is always sorted — **Name ascending** is the default. When sorting by any other column,
ties break by **name (A→Z)** — so e.g. repos sharing a branch list alphabetically within that
branch — and the name tiebreak stays ascending even when the column is sorted descending.

| Key | Sort by |
|-----|---------|
| `n` | Name |
| `c` | Checked-out branch |
| `s` | Status |
| `a` | Ahead / behind |
| `d` | Dirty count |
| `l` | Last-commit time |
| `w` | Worktree count |
| `b` | Branch count |
| `k` | Stash count |
| `p` | Commits pulled |
| `g` | Files changed |

Re-picking the same column (or re-clicking its header) flips the direction. The sort
persists across runs. The sort menu lists **only the columns currently visible** — toggle a
column off and its sort option disappears too.

:::note[Worktree vs branch glyph]
Worktrees render `⑃N` (cyan) and branches `⑂N` (green) — distinguished by both the glyph
(inverted vs upright fork) and the color.
:::

## Lazy loading

The git-derived columns (`a`/`d`/`l`/`b`/`s`) fetch per-repo details in the background the
first time one is enabled — cells show `…` until the data is ready. The worktree column
(`w`) is free, since worktrees are already discovered during the initial scan. The pulled
(`p`) and changed (`c`) columns come straight from each pull, so they need no extra fetch —
they show `…` only while a repo is still pulling.
